1. Non-negotiable defects summary

These are the mandatory fixes.

Defect A — confidence-weighted scoring is absent

The engine currently treats noisy, stale, sparse, and high-confidence data too similarly.

This is unacceptable.

Required:
	•	all three score layers must support confidence-weighted metrics
	•	especially Resource and Seasonal

Defect B — strength output is still flat

The engine still accepts a flat strength_output_index instead of deriving weighted session output from exercise-level data.

This is unacceptable.

Required:
	•	add exercise-level model
	•	compute weighted session strength output
	•	expose primary compound vs accessory contribution

Defect C — carb adequacy is falsely labeled

Current carb adequacy logic is just average daily carb adherence.
That is not “carb adequacy around training.”

Required:
	•	compute peri-training carb adequacy
	•	tie targets to day type

Defect D — fat oscillation is inferred from recommendations instead of intake

Current logic rewards recommended oscillation instead of confirmed actual oscillation.

Required:
	•	score actual intake behavior first

Defect E — deload compliance is fake

Counting reset labels is not deload compliance.

Required:
	•	compute actual reduction in stress structure

Defect F — monotony logic is too crude

Current monotony logic is an oversimplified label dominance proxy.

Required:
	•	separate cardio, lift, and macro monotony channels

Defect G — seasonal layer is incomplete

Light exposure consistency is absent.
Virility trend is underbuilt.

Required:
	•	implement both properly

Defect H — nutrition delta allocation is too blunt

Current ingredient adjustment logic is too concentrated and under-constrained.

Required:
	•	impose caps and distribution rules

B. Replace fake deload compliance

Current logic:
	•	based on reset/resensitize day count

This must be removed.

Required replacement

Compute a real deloadScore28d.

Suggested formula:

DeloadScore =
0.30 \cdot VolumeReductionScore +
0.20 \cdot NeuralReductionScore +
0.20 \cdot Zone2IncreaseScore +
0.15 \cdot SurgeReductionScore +
0.15 \cdot LiftStrainReductionScore

Where each subscore is 0–100.

Then:
	•	deloadCompliance28d = deloadScore28d >= 70

Required sub-calculations
Volume reduction
Compare final 7 days of 28-day window to prior 21-day baseline or prior 14-day baseline.

VolumeRatio = \frac{LiftVolume_{last7}}{LiftVolume_{prior14}}

Map:
	•	<= 0.65 → excellent
	•	0.66–0.80 → good
	•	0.81–0.95 → partial
	•	0.95 → poor

Neural reduction
Compare neural session count in final 7 days vs prior 14 days.

Zone 2 increase
Check whether Zone 2 share rises in final 7 days.

Surge reduction
Check whether surge macro-day frequency drops.

Lift strain reduction
Check whether average lift strain drops.

This is non-negotiable.

FILE: app/vitals_engine.py

Current role

Recommendation engine and orchestration layer.

Current problem

Required actions

⸻

A. Replace score consumption with weighted score results

The engine must consume weighted outputs from vitals_scoring.py.

Do not keep parallel raw-only score paths.

⸻

B. Tighten cardio decision logic

Current cardio selection is too flat.

Required rule stack:

Hard-stop override
Force Zone 2 or recovery if any:
	•	HRV < 90% of 7d average
	•	RHR > 7d average + 4 bpm
	•	sleep < 360 min
	•	composite < 55

Distribution cap
If zone3_count_7d >= 3, default to Zone 2 unless:
	•	composite >= 90
	•	no fatigue flags

Sequential brake
If previous two completed cardio sessions were Zone 3:
	•	today defaults to Zone 2

Resensitize cap
On cycle days 22–28:
	•	max 1 Zone 3 session per rolling 7d unless composite exceptionally high and fatigue clear

This logic must be explicit.
Not implied.

⸻

C. Tighten lift decision logic

Maintain class-based lift assignment, but add braking rules:
	•	if primary compound trend is negative and fatigue flags present, downgrade neural assignment
	•	if composite high but prior 2 neural sessions already occurred in rolling 7d, consider hypertrophy build instead
	•	resensitize week must cap neural frequency

⸻

D. Macro day recommendation must remain recommendation, not evidence

The engine must stop treating recommended macro day as proof of achieved oscillation.

Recommendation is output only.
Behavior scoring must depend on actual intake data.

⸻

E. Ingredient adjustment logic must be constrained

Current adjustment routing is too blunt.

Required safeguards:

Per-ingredient cap
No single ingredient absorbs more than a configurable percentage of the day’s macro delta.

Per-window cap
No single feeding window absorbs too much total delta.

Protected windows
Pre/post training windows must maintain intended digestion structure.

Distributed adjustment
Macro delta should be spread across ranked candidate ingredients until caps reached.

Implement a proper allocation algorithm, not first-hit dumping.

⸻

F. Add confidence summaries to dashboard/recommendation response

Engine response must expose:
	•	overall score confidence
	•	low-confidence contributors
	•	warnings like:
	•	“FFM trend confidence low”
	•	“waist stale”
	•	“HRV sparse”

Without that, the user cannot interpret whether the engine is confident or guessing.
