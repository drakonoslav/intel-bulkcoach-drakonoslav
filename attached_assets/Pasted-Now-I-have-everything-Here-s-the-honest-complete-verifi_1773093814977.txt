Answering your 5 questions honestly:

1. Source-of-truth mapping
What is stored: The four catalog-level fields (primary_muscles, compound_or_isolation, slots, equipment_tags) bundled into a single JSONB blob under the column name biomechanics in intel_exercise_mapping.
Is it true biomechanics? No. It is the muscle activation matrix and catalog metadata from Intel — not biomechanics in the sense you described (implement type, body position, laterality, resistance direction, etc.). The column name biomechanics is a misnomer. These are catalog/muscle fields, not biomechanics metadata.
2. Set-level persistence
The intel_exercise_id INTEGER column was added to strength_sets. Future game sets will retain both local_exercise_id (resolved) and intel_exercise_id (raw Intel ID).
Manual/custom exercises will have intel_exercise_id = NULL, which is correct.
Existing game sets written before the migration also have intel_exercise_id = NULL since the column didn't exist. There are currently 0 game sets in the database.
3. Receipt/query enrichment
GET /api/game-exercise-sets — now JOINs mapping to return intel_exercise_id, intel_exercise_name, and the stored catalog blob.
GET /api/strength-sets — same JOIN.
GET /api/intel-receipts/:date — now returns exerciseBiomechanics lookup alongside the receipt.
But the data labeled "biomechanics" in all three routes is actually catalog metadata, not true biomechanics.
4. Non-destructive guarantee
All changes are strictly additive: new column on intel_exercise_mapping, new column on strength_sets, wider SELECT with LEFT JOIN. No existing insert/update/delete logic was modified. The only change in persistence was adding $4 to the game-set INSERT column list.
