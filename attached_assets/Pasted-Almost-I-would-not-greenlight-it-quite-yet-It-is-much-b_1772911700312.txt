Send Intel this addition:

2. Add idempotency / dedupe rules for POST /game/log-set

This is critical.

Expo may retry requests. If Intel logs duplicate bridge sets, freshness and fatigue will drift upward incorrectly.

You need:
	•	a client-generated event id or log id
	•	idempotent write behavior
	•	a statement of what happens if the same set is sent twice

Tell Intel:

Add idempotency rules for /game/log-set.
I want a client event ID in the request and dedupe behavior documented so repeated submissions do not double-count fatigue/load.

3. Clarify whether session-close changes state or is just a finalizer

You want to know whether:
	•	all important game-state updates happen at each log-set
	•	or whether some state is only finalized at session-close

Best practice here is:
	•	log-set updates state incrementally
	•	session-close is a finalizer/summary marker, not required for correctness if the app crashes late

Ask Intel to state that clearly.

4. Refine the freshness scale explanation

The formula is okay for v1 bridge integration, but it needs one sentence explaining interpretation.

freshness = 1 / (1 + fatigue / 1000)

That means freshness is a transformed index, not a physiological percentage.

Expo should not assume:
	•	0.80 = 80% recovered
	•	0.30 = 30% recovered

It is just a bounded readiness-like scalar.

So Intel should label it as:
	•	normalized freshness index
	•	0 to 1
	•	derived from fatigue, not literal recovery percentage
